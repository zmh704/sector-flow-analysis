> 生成于 2026-07-14。本文档基于对全部核心源码的逐文件精读(calc.js / leaders.js / config.js / data.js / app.js / modals.js / charts.js / analyze.js / reprocess.js / generate-list.js),所有发现均给出具体位置和依据,按严重程度排序。

## 〇、一个贯穿性根因(先读这个)

**「涉及股票」字段以格式化字符串存数值**(格式 `简称(代码|成交额|净额|涨跌幅|成交量)`,其中成交额/净额是 `50.41亿`、`-5.00亿` 这类带单位文本)。多处代码直接用 `parseFloat` 读这些字符串,而 **`parseFloat("50.41亿") = 50.41`、`parseFloat("9000万") = 9000`** —— 单位被静默丢弃。这直接造成下面 P0-1 的比较错误,也是 analyze.js 侧健壮性风险(P2-1)的同源问题。

**根治方向**:在解析入口(`parseStocks` / `_stockFieldIndex` 构建处)一次性把金额归一化为统一数量级的数字,下游全部用归一化后的值,而不是在每个比较点打补丁:

```js
function toWan(s) {            // 统一归一化到「万」
  const v = parseFloat(String(s).replace(/,/g, ''));
  if (isNaN(v)) return NaN;
  if (/亿/.test(s)) return v * 10000;
  return v;                    // 万 / 手 视为基准
}
```

## 一、P0 — 确定的正确性 bug(影响分析结论)

### 1. 放量/缩量判断跨单位比较错误 ⚠️ 核心筛选逻辑失真
- **位置**:`calc.js:184-188`、`calc.js:204-206`(成交额比较);`calc.js:163`、`calc.js:220-225`(成交量比较)
- **问题**:今日推荐的条件 C/D/H 直接 `parseFloat` 涉及股票里的带单位字符串再乘阈值比较。相邻两日单位档位不同时(亿↔万)结论必错:`parseFloat("1.2亿")=1.2` vs `parseFloat("9000万")=9000`,"放量到 1.2 亿"被判成大幅缩量。
- **影响**:今日推荐的放量/缩量过滤在单位切换的股票上**系统性误判**,直接影响推荐结果。
- **修复**:按上面的 `toWan` 在解析阶段归一化。注意:只看正负号的判断(如 `主力净额 > 0`)不受影响,不必改。

### 2. 连续流入天数对异常净额虚增
- **位置**:`calc.js:254`(`calcConsecutiveInflow`)vs `calc.js:346`(`condNetPositive`)
- **问题**:同样是"净额 > 0"语义,两处对 NaN 的处理相反:
  ```js
  if (!sector || Number(sector.主力净额) <= 0) break;  // NaN<=0 为 false → 不中断,该日被计入连续天数
  return Number(sector.主力净额) > 0;                   // NaN>0 为 false → 排除
  ```
  某天板块净额字段损坏(空串/带单位字符串)时,连续天数**虚增**,而入选判断又把同一板块排除,自相矛盾。
- **修复**:统一为"非正即断":`const net = Number(sector.主力净额); if (!sector || !(net > 0)) break;`

### 3. 跨年时「最近 12 个文件」会选错
- **位置**:`data.js:233-238`(选取)配合 `data.js:19-33`(`toDateNum`)
- **问题**:`loadAllJsonFiles` 在数据未加载时用 `toDateNum(extractDateLabel(a))` 排序取 `slice(-12)`,但 `toDateNum` 的跨年推断只在传入 `genTime` 时生效,此处没传 → 所有文件按当前年算。1 月运行且目录里有去年 12 月文件时,`12月xx日` 排在 `1月xx日` 之后,**最新的 1 月文件被丢弃、加载的是旧文件**。
- **修复**:文件名保留年份,或按服务端 `/api/list` 已有的 mtime 顺序直接取尾部 12 个,不要在前端重新按"月日"排序。

### 4. 概念板块流入/流出单选疑似漏绑事件(需确认)
- **位置**:`app.js:9-14`
- **问题**:行业单选 `flowFilterIndustry` 绑定了 `change → updateCharts`,紧随其后只有"概念流入/流出单选"的**注释**,没有对应 `flowFilterConcept` 的绑定代码。若其它文件也未绑定,切换概念图表的流入/流出不会刷新。
- **验证**:全局搜索 `flowFilterConcept`;若确实无绑定,补一行与行业对称的绑定即可。

### 5. Excel 重复行导致「股票数量/总额」与「涉及股票」口径不一
- **位置**:`analyze.js:89-90` / `:94-95`
- **问题**:`涉及股票` 按字符串去重,但 `count`、`totalNet`、`totalTurnover` 对每一行无条件累加。源表有重复行时:数量计 2、金额双计,列表只显示 1 只。
- **修复**:把累加移进 `!stockSet.has(stockStr)` 分支内,三个口径统一。

## 二、P1 — 性能与内存(「用久变慢」的直接原因)

### 1. TradingView 组件反复新建、从不销毁 ⚠️ 内存持续增长的真凶
- **位置**:`modals.js:817-889`(`loadTrendStock`,重点 `:827`、`:863`)
- **问题**:每次切换个股/数据源都 `container.innerHTML = ''` 后 `new TradingView.widget(...)`。`innerHTML=''` 只摘掉 DOM,**widget 实例和它挂在 `window` 上的 `message` 监听器、内部定时器不会被清理**。反复打开趋势弹窗、切股票、切"新浪⇄TradingView",widget 持续累积。
- **影响**:页面越用越卡、内存持续上涨 —— 这是长时间使用后变慢的最主要嫌疑(Chart.js 侧反而全部正确 `destroy()` 了,见"做得好"一节)。
- **修复**:
  ```js
  if (window._tvWidget) { try { window._tvWidget.remove(); } catch(e){} window._tvWidget = null; }
  window._tvWidget = new TradingView.widget({...});
  ```
  并在 `closeTrendModal`(`modals.js:733`)中同样 `remove()`。

### 2. 「查看全部」弹窗配对计算的重复劳动
- **位置**:`modals.js:175-191`(`showAllData` 配对)
- **问题**:N×M 双层循环中,内层每轮重建同一批 `otherStocks` Set,且对同一板块重复调用 `calcConsecutiveInflow`(要回扫历史,是重活)。板块上百时点开弹窗有可感知卡顿。
- **修复**:进入外层循环前对 `otherList` 预构建一次 `{name, stockSet, days}`,内层只做集合交集。

### 3. 一次刷新中 `filterSectors` 反复重建日期 Map
- **位置**:`calc.js:389-409`、`getFocusSectors` `calc.js:265-274`
- **问题**:一次完整刷新里 `filterSectors` 被调用 4+ 次(推荐区行业/概念 + 关注区行业/概念 + 加星集合),每次重建 curr/prev/prev2/prev3 四个 Map、重复 `sortDateFileList()`。
- **修复**:按 `type + currentDateFile` 缓存结果,挂进现有的 `invalidateDateCaches` 失效体系(`config.js:95-112`)——不要另起炉灶。

### 4. `/api/parse` 同步解析阻塞整个服务
- **位置**:`analyze.js:51-53`(`sheet_to_json` + 全表循环),被 `server.js /api/parse` 在请求内直接调用
- **问题**:单进程 http 服务器里跑 CPU 密集同步解析,上传大 Excel 期间**所有请求挂起**(含 `/api/list`)。
- **修复**:本地单人使用影响有限,可先只加上传体积上限;若要根治用 `worker_threads` 跑 xlsx 解析。注:`reprocess.js` 的同步 IO 是一次性 CLI 批处理,合理,**不用改**。

### 5. 搜索框无防抖,每键整表重建
- **位置**:`modals.js:28-131`(`renderModalTable`,过滤在 `:44-54`)
- **问题**:每敲一个字符就 filter → sort → 整表重建。已用 `DocumentFragment` 但无防抖,中文输入法逐字触发掉帧。
- **修复**:给搜索输入加 200ms debounce(项目里已有 `debounce` 工具,`config.js`)。行数量级不大,虚拟化没必要。

### 6. 新浪 K 线图片强制破缓存
- **位置**:`modals.js:829-851`
- **问题**:图片 URL 拼 `?ts=Date.now()`,同一只股票反复查看每次都重新下载最多 4 张 K 线图。
- **修复**:常规切换不加时间戳(复用浏览器缓存),仅手动刷新时加。

### 7. 数据加载无重入保护,存在竞态
- **位置**:`data.js:197-284`(`loadAllJsonFiles`),触发点 `app.js:277`(onload)与 `app.js:366`(Excel 上传后)
- **问题**:分批 `await` 的长流程直接读写全局状态。首屏加载未完成时上传 Excel 会再次进入,第二次的 `resetLoadedData()` 清掉进行中的第一次结果,两个循环交错写 `dateFileList`。
- **修复**:模块级 in-flight promise,重入时复用:
  ```js
  let _loadingPromise = null;
  async function loadAllJsonFiles() {
    if (_loadingPromise) return _loadingPromise;
    _loadingPromise = (async () => { /* 原函数体 */ })();
    try { return await _loadingPromise; } finally { _loadingPromise = null; }
  }
  ```

## 三、P2 — 健壮性与安全

### 1. analyze.js 裸 `parseFloat` 读 Excel 数值列(健壮性风险)
- **位置**:`analyze.js:74-77`
- **问题**:当前数据源的数值列是纯数字时工作正常;但一旦换成带单位/千分位文本的导出格式(`"50.41亿"`、`"5,041,234"`),`parseFloat` 静默截断且 `|| 0` 吞掉错误,金额缩小几个数量级,**核心排序结论全毁且无任何告警**。
- **修复**:同"贯穿性根因"的单位感知解析函数;至少在解析到可疑字符串时告警而非静默归零。

### 2. detectColumns 反向 `includes` 过宽 + 无"已占用列"去重
- **位置**:`analyze.js:39-43`
- **问题**:`n.includes(String(c))` 反向包含让短列名(「额」「量」)命中多个候选;且同一物理列可被多个逻辑键认领,错配是**静默**的。代码注释想靠"候选互不重叠"防误配,反向 includes 恰好绕过了这层防护。
- **修复**:去掉反向包含,只留 `c.includes(n)`;维护 `used` Set,一列只分配一次。

### 3. 唯一的 XSS 缺口:新浪图表分支的 `stockCode`
- **位置**:`modals.js:829-851`
- **问题**:全项目 `escapeHtml` 用得相当到位,**唯独**这里把来自 Excel 的 `stockCode` 未转义拼进 `innerHTML` 的 `<img src>`。正常股票代码是数字风险低,但脏数据带引号/尖括号即可注入。
- **修复**:改用 `document.createElement('img')` + `img.src = ...` 赋值(属性赋值天然安全),顺手把 P1-6 的缓存问题一起改了。

### 4. `package.json` 未声明 `xlsx`
- **位置**:`analyze.js:5`、`reprocess.js:5` 均 `require('xlsx')`,package.json 只声明 `chart.js`
- **问题**:当前靠 node_modules 里已装的 xlsx 才能跑;换机器 `npm install` 或 `npm ci` 后 server.js/reprocess.js 直接 `Cannot find module 'xlsx'` 崩溃。
- **修复**:`npm install xlsx --save`。(注:此前旧文档称"会导致 Excel 解析完全无法工作"不准确——本机实际已安装,能正常工作,缺的是声明。)

### 5. `parseStocks` 对内嵌逗号/右括号不健壮
- **位置**:`calc.js:16-17`
- **问题**:`split(',')` + `\(([^)]+)\)` 假设字段内无逗号无右括号;若成交额出现千分位(`1,234万`)整条解析错乱,无括号项被静默丢弃。
- **建议**:数据源可控则加注释标注"依赖上游格式";不可控则改为整体正则逐条匹配。

### 6. generate-list.js 按 mtime 排序 ≠ 交易日顺序
- **位置**:`generate-list.js:29-36`
- **问题**:JSON 是 reprocess 按目录枚举顺序写出的,mtime 反映**写入先后**而非交易日。重跑 reprocess 或补录历史数据后,静态部署(list.json 回退模式)的时间轴会乱序。顺带:比较器里反复 `statSync` 被调 O(n log n) 次,应先 map 出 mtime 再排序。
- **修复**:按文件名中的 `月日` 解析排序(带跨年推断),与 P0-3 的修复统一处理。

## 五、做得好的地方(改的时候别破坏)

- **Chart.js 实例管理正确**:`trendNetChart`/`trendTurnoverChart` 在 4 处成对 `destroy()+置null`(`modals.js:586-587`、`:618-619`、`:737-744`、`:252-254`);主图走增量 `update()` 复用(`charts.js:84-91`)。图表侧**没有**内存泄漏。
- **切换日期零网络请求**:数据一次性缓存在 `allDataByDate`,日期切换只改指针 + 重渲染。
- **缓存失效分层设计**(`config.js:95-112`):`invalidateAllCaches`/`invalidateDateCaches` 区分全量索引与随日期变化的缓存,失效点覆盖完整。新增缓存(如 P1-3)应挂进这套体系。
- **`calcStockConsecutiveDays` 懒加载 + 30 天截断**(`calc.js:101-141`):按需构建每日 Map,防止无限回溯,不要改成一次性全量构建。
- **`buildSectorMap` O(1) 查找 + 条件函数独立可开关**:`passesLeaderConditions` 各条件拆成独立方法、注释开关,设计清晰。
- **`escapeHtml` 覆盖面好**:innerHTML 路径基本处处转义(唯一例外见 P2-3)。
- **localStorage 读写包 try/catch**(`config.js:116-134`),解析失败回退空数组。
- **server.js 路径遍历防护**用 `path.relative` 判断,写法正确。
